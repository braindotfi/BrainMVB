=========================================================================
INTEGRATION PRINCIPLES (these define "production" here)
=========================================================================
1. ACTOR = SESSION. The current member is whoever is authenticated. The platform never
   sends an actor field on session-authenticated calls (core strips it anyway, per the
   contract). There is NO member switcher of any kind — demoing two approvers means two
   signed-in sessions (two browsers/profiles).
2. ENFORCEMENT LIVES IN CORE ONLY. Do not port, keep, or re-implement authorizeApproval or
   the six checks client-side. Delete any mock gate if present. UI affordances (hiding
   approve controls from a viewer, disabling above-limit approvals) are derived from the
   CURRENT MEMBER'S envelope fetched from core (GET /v1/members/{self} or the session's
   member payload) — they are conveniences, and the server's response is always the final
   word. A user who bypasses the UI still hits core's 403.
3. REJECTIONS ARE CORE'S. Every 403 { error, reason, detail } renders as an inline #D20344
   notice mapping the exact contract reason string to one plain-language sentence:
   actor_unresolved | actor_inactive | domain_not_authorized | actor_limit_exceeded |
   second_approval_required | self_approval_blocked | last_admin_protected.
   Unknown reason → generic "Not permitted" + the raw reason in Technical Detail. Never a
   silent no-op, never a dead tap.
4. STATE COMES FROM CORE. Proposal status (including "awaiting_second_approval"), the
   approvals array, actor verification — all read from core responses/webhooks; the platform
   holds no parallel state machine. Handle network failure gracefully: show the last known
   state + a retry affordance; never optimistically mark approved before core confirms.

API WIRING (per the contract)
=========================================================================
- GET /v1/members (list; filter role/domain), GET /v1/members/{id}
- POST /v1/members, PATCH /v1/members/{id}, DELETE /v1/members/{id} (deactivate) — render
  these mutations ONLY for admins per the current member's role, but rely on core's
  admin-gating as the real check.
- POST/DELETE /v1/members/{id}/identity-links
- Approve path: existing proposal approve call, now consuming the contract response
  { status, approvals: [{memberId, at, verification}], actor: {memberId, verification} }.
- Webhooks/subscriptions: member.changed → refresh members views;
  proposal.awaiting_second_approval → update the queue row live.
- Every member mutation confirms in plain language BEFORE the call (persistent-config
  change discipline), then reflects core's response — including surfacing
  last_admin_protected as a disabled control with caption "At least one admin is required"
  when core rejects, and proactively disabling it when the fetched member list shows only
  one active admin.

   second_approval_required rejection in plain language. A distinct qualified member's

   and authority line populated from core data ("sarah@meridian approved · approver · within
   her $10,000 limit"). Viewers see no approve controls (derived from role) and core rejects
   them regardless.

   deactivated/unknown member: "(member inactive)" — deactivated members must still resolve
