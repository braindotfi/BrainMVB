Fix the root cause behind unmapped brain-core actions (like raw.ingest.new) rendering as red "FLAGGED" cards with Approve/Reject buttons and generic, inaccurate copy in the Inbox and Audit Log.
Root cause: /audit/events already returns authoritative event_type (system_activity | assistant_activity | flagged), category, and severity fields on every event — brain-core defaults unset events to system_activity. The client currently ignores all three fields entirely. Instead, classify() in client/src/lib/brainAudit.ts re-derives its own category purely by pattern-matching the action string against a local ACTION_MAP, and defaults anything not in that map to flagged. Since raw.ingest.new (routine data ingestion, not a decision) isn't in the map, it falls through to flagged, gets routed into Inbox's Needs Review tab, and picks up a fabricated generic "why" line that doesn't describe what actually happened. This will keep happening for every future brain-core action the client hasn't manually special-cased.
Changes needed:

Add event_type, category, and severity to the BrainAuditEvent interface in brainAudit.ts — they're already on the wire, just not declared client-side.
In classify(), use event.event_type from brain-core as the primary signal for whether something is flagged vs. informational, instead of deriving it solely from the local ACTION_MAP. The ACTION_MAP can still provide nicer human-readable summaries per action (e.g. "Payment approved" vs. the raw action id), but it should no longer be the thing deciding risk classification — that's brain-core's job now that it exposes it.
Flip the fallback default for unmapped actions from flagged to a non-actionable classification, matching brain-core's own default of system_activity. Add a system_activity-equivalent bucket to the client's local AuditEventType type if one doesn't already fit, alongside the existing assistant_activity handling pattern (isAssistantActivity / ASSISTANT_SUBTYPES) — generalize that mechanism to cover system_activity broadly rather than a hardcoded per-action allowlist, since the point of reading brain-core's real field is to stop needing to hand-maintain that list at all.
In InboxPage.tsx, exclude system_activity events from all Inbox tabs the same way assistant_activity events are already excluded (the existing if (isAssistantActivity(r)) continue; pattern) — routine system activity like data ingestion doesn't belong in a queue built around decisions, only in Audit Log.
Fix the "why" text generator (auditWhy in InboxPage.tsx) so it never falls back to the generic "Flagged for attention — Brain saw something that didn't fit the usual pattern" line for events that aren't genuinely risk-classified. For any event still legitimately flagged but without a specific mapped summary, prefer showing the honest raw action id (already done in brainAudit.ts's classify() fallback) over fabricated boilerplate — the two functions should agree, not diverge.
Once this lands, add human-readable summaries specifically for raw.ingest.new and raw.ingest.deduplicated (e.g. "New data ingested — Brain pulled in new records to process" / "Duplicate data — already ingested previously, skipped"), sourced from the actual event, not generic text.

Scope: this touches client/src/lib/brainAudit.ts, client/src/lib/auditTypes.ts, and client/src/pages/InboxPage.tsx only — specifically the event classification, tab-routing, and why-text logic. No other pages or unrelated logic should change.
