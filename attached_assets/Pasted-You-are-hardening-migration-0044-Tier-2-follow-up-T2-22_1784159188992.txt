You are hardening migration 0044 (Tier 2 follow-up T2-22/T2-23) so it cannot fail-closed
a deploy on pre-existing duplicate rows. Migration
services/ledger/migrations/0044_ledger_direct_write_dedup_constraints.sql adds three
partial UNIQUE indexes but includes NO dedup of existing rows, so it aborts the deploy
(`could not create unique index ... is duplicated`) in any environment where the old
racy SELECT-then-INSERT path already produced duplicate counterparties or obligations.

WHY EDIT 0044 RATHER THAN ADD A NEW MIGRATION: migrations run in order and the runner
aborts on the first failure, so a migration numbered after 0044 would never run in the
exact environment that needs it (0044 fails first). Editing 0044 is safe because the
runner skips already-applied migrations by name/version: an environment where 0044
already succeeded had no duplicates (the added dedup is a no-op there and will not
re-run), and an environment where 0044 failed has not recorded it as applied (the
edited version applies cleanly). Confirm this runner behavior in tools/migrate/src/
runner.ts before proceeding; if the runner does NOT skip-by-name (e.g. it hashes files
and would re-run or error on a changed applied migration), STOP and report that instead
of editing.

FIX - inside 0044, BEFORE each CREATE UNIQUE INDEX (same transaction), add a
deterministic, idempotent dedup/merge step:

1. Duplicate counterparties (group by owner_id, normalized_name, type WHERE
   normalized_name IS NOT NULL AND canonical_counterparty_id IS NULL): choose a
   deterministic survivor (e.g. MIN(id), or earliest created_at then MIN(id) as a
   tiebreak), then BEFORE deleting the losers, REPOINT every foreign-key reference to a
   loser onto the survivor. Enumerate the referencing tables from the schema (at least
   ledger_obligations.counterparty_id, ledger_payment_intents.destination_counterparty_id,

   them all. Merge alias/metadata arrays onto the survivor (union, distinct). Then delete
   the losers.

2. Duplicate obligations (group by owner_id, counterparty_id, type, amount_due, currency,
   due_date WHERE external_key IS NULL): same pattern. Deterministic survivor, REPOINT
   references (at least ledger_payment_intents.obligation_id, ledger_reconciliation_matches

   distinct) onto the survivor, then delete the losers.

3. Only after both dedups, run the existing CREATE UNIQUE INDEX statements unchanged.

Do NOT collapse two rows that are NOT actually duplicates: the grouping key is exactly
the unique-index key, so any group with >1 row already violates the constraint the index
is about to enforce and is by definition a duplicate under this contract. Preserve the
richest survivor (merge arrays) rather than dropping data.
