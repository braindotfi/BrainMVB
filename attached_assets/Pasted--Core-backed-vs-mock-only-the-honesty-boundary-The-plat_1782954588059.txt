## Core-backed vs mock-only (the honesty boundary)

The platform is a reference client. CORE-BACKED capabilities are enforced by brain-core and
may be stated as guarantees. MOCK-ONLY capabilities are product design the platform simulates
ahead of core ("core leads" — every mock-only row is core backlog, not a permanent state).
Every mock store or simulated behavior must carry a file-header comment declaring its side.
Update this table whenever a row changes state. Transition rule: a row moves to CORE-BACKED
only when the platform consumes core's real API for it AND the corresponding mock layer has
been deleted — not merely when core merges.

| Capability | Status | Source of truth |
|---|---|---|
| Propose → approve → execute pipeline (propose-only; Brain never moves funds) | CORE-BACKED | brain-core |
| Fraud guards (bank-detail change, duplicate) escalate; boot-fenced | CORE-BACKED | brain-core (check-invariants) |
| Audit records + on-chain anchoring (batched, Base) | CORE-BACKED | brain-core Audit layer |
| MCP propose-only entry (agents propose, never approve/execute) | CORE-BACKED | brain-core |
| Members, roles, approval limits, second approval, actor≠payee | CORE-BACKED | brain-core authorizeApproval per docs/contracts/members-attribution.md; platform renders, never enforces |
| Actor attribution + verification strength | CORE-BACKED | brain-core resolveActor; platform sessions are "session"; idp_verified stubbed in core, no flow yet |
| Vendor trust statuses gating automation (new/known/trusted/under_review) | MOCK-ONLY | platform mockVendors — core contract pending (next in "core leads" backlog) |
| Rules as standing delegation (auto-clear under policy) | MOCK-ONLY | platform rulesStore — core contract pending (with vendor trust) |
| Rule suggestions from tenant history (evidence + confidence) | MOCK-ONLY | platform mock — real derivation from Ledger/Audit pending |
| Anchor pending→anchored progression | CORE-BACKED | batching real; platform must never show "anchored" before the batch settles |

Demo rule of thumb: CORE-BACKED rows may be stated as guarantees ("a member cannot approve
their own payment — core rejects it on every surface"). MOCK-ONLY rows are stated as product
design, never as current enforcement.
