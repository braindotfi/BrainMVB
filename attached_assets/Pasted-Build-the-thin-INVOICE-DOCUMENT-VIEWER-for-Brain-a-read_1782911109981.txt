Build the thin INVOICE / DOCUMENT VIEWER for Brain: a read-only popup that shows the source
document Brain ingested, the fields it extracted, and where the document came from. It opens
when a user taps an invoice/document reference (e.g. the "Linked · Invoice" row on an audit
record or proposal detail). This is a VIEWER, not an AP system.

STYLING — MATCH THE CURRENT APP EXACTLY. Reuse existing tokens, fonts, spacing, radii, card/
row/chip/button styles, and the proposal-detail / audit-record popup shell. Dark indigo
canvas, purple accents, monospace for amounts/ids/dates. #D20344 for alerts/mismatch only;
success/green for a clean match; purple for interactive. NO orange (reserved for audit tabs).

SCOPE — THIN. Build ONLY: the invoice viewer popup (source document pane, extracted fields,
provenance, amount-coherence note), the openInvoiceDetail helper, and the split linked rows.
Do NOT build invoice creation, editing, approval routing, AP aging, or any invoice management
— those live in the customer's accounting system. Brain reads the document; the accounting app
owns it. Reinforce this with a caption in the viewer.

WHERE INVOICE DATA COMES FROM (frame the model around this): Brain does not originate invoices
— it INGESTS them via connectors (accounting sync like QuickBooks/Xero/Merge, or an email/
document channel). The original artifact lives in the Raw layer; the parsed record in the
Ledger layer. So an invoice is: a source document (PDF/image ref) + extracted structured
fields + provenance (which connector, when, a Provenance enum value) + a ledger ref. Records
reference an invoice BY ID (like rules and vendors) — never by a copied string.

DATA MODEL:
type Provenance = "CONNECTOR_SYNC" | "EMAIL_INGEST" | "DOCUMENT_UPLOAD" | "PAYMENT_RAIL";
interface InvoiceLineItem { label: string; amount: number; }
interface Invoice {
  id: string;                    // "AWS-2026-07" (the invoice number doubles as id here)
  vendorId: string;              // resolves to a Vendor via openVendorDetail
  vendorName: string;
  amount: number;                // total
  dueDateLabel: string;
  payeeAccountLast4: string;
  lineItems: InvoiceLineItem[];
  documentHref?: string;         // link to the original PDF/image (Raw layer)
  provenance: {
    source: string;              // "QuickBooks (accounting)"
    ingestedAtLabel: string;     // "Jul 6, 3:12 PM ET"
    enum: Provenance;            // "CONNECTOR_SYNC"
    ledgerRef: string;           // "LGR-88214"
  };
  proposalId?: string;           // the proposal that reasoned over this invoice
}
- Add `invoiceId?: string` to the relevant records (proposals, AuditRecords for payment
  events, settled Approved Record cards) so their linked section can resolve the invoice.

INVOICE VIEWER POPUP (same shell as other detail popups), top to bottom:
- Header: "Invoice" chip + invoice id in mono, close X.
- "Source document": a rendered representation of the document on a LIGHT (paper-like) pane —
  vendor letterhead, line items, total, due date — with an "Open original PDF" button
  (documentHref). This pane depicts paper, so a light background here is intentional and
  allowed even in the dark app.
- "What Brain extracted": a mono block — vendor, invoice id, amount, due date, pay-to
  account ··{last4}. This lets the user confirm Brain parsed the document correctly.
- AMOUNT-COHERENCE NOTE: if the invoice amount equals the linked proposal/payment amount, a
  green success line "Amount matches the approved payment ($X)." If it does NOT match, a
  #D20344 line "Amount differs from the payment ($invoice vs $payment) — review." (This makes
  the coherence check visible, and is the guard against a resolves-but-lies mismatch.)
- "Where it came from": a mono provenance block — source, ingested-at, provenance enum,
  ledger ref.
- Caption at the bottom: "A viewer, not an AP system — Brain reads the document, your
  accounting app owns it."
