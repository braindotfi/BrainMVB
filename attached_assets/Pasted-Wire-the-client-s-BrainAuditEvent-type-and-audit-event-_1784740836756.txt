Wire the client's BrainAuditEvent type and audit event mapping to brain-core's actor_ref field so actor resolution actually works, instead of defaulting to omitting the actor whenever the raw actor string looks like a machine ID.
Background: brain-core's /audit/events endpoint returns an actor_ref object on every event — { id, type, display_name?, email?, lookup? } — where display_name/email are present when the emitting service captured them inline, and lookup is a relative resolution path (/v1/members/{id} for user actors, /v1/agents/{id} for agent actors) when it wasn't. Right now the client's BrainAuditEvent interface (client/src/lib/brainAudit.ts) only declares actor: string and never reads actor_ref at all, so even when brain-core can resolve an actor, the client can't see it and silently omits the actor line.
Changes needed:

Extend the BrainAuditEvent interface to include the actor_ref field brain-core already returns ({ id: string, type: string, display_name?: string, email?: string, lookup?: string }).
In mapAuditEventToRecord, prefer actor_ref.display_name or actor_ref.email when present, instead of only falling back to humanReadableActor(event.actor) (which will keep correctly filtering raw IDs, but should now be the last resort, not the only path).
When display_name/email are absent but actor_ref.lookup is present, resolve it via a fetch to that path (/v1/members/{id} or /v1/agents/{id}, proxied through the existing BFF passthrough the same way /audit/events already is) and cache the result — don't refetch the same actor on every render or every record in a list.
If resolution fails or lookup is absent and no inline display data exists, keep the current honest fallback: omit the actor line entirely. Never show the raw user_.../agent_... string as a substitute.
Apply this consistently everywhere actor is rendered — the Audit Log list, the Audit Record popup, and the Inbox card for any assistant-activity or flagged items that show an actor.

Note: some of brain-core's actor-lookup endpoints (/v1/members/{id}, /v1/agents/{id}) may require platform-level auth and need to go through the BFF rather than being called from the browser directly — confirm this against how /audit/events is currently proxied in server/brain/proxy.ts before wiring the fetch, since a few adjacent endpoints in brain-core's surface are known to require X-Platform-Service-Auth rather than the session/member token.
Scope: this only touches the audit event type/mapping (brainAudit.ts, auditTypes.ts if the AuditRecord actor field needs to carry resolved display data) and wherever actor is rendered in Audit Log / Inbox. No other pages or unrelated logic should change.
