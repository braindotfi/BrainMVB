Application Flow:

1. **Agent Type Selection Screen**
   - Six cards representing each agent type (Trading, Lending, Yield, Payments, Analytics, Custom)
   - Each card shows icon, description, and key use case
   - On selection, navigate to the configuration form for that type
   - Include a "Learn More" tooltip for each type

2. **Dynamic Form Generation**
   - Render form fields based on selected agent_type
   - Group related fields into collapsible sections (e.g., "Position Controls" for Trading agents)
   - Add inline hints and tooltips for each input
   - Validate inputs in real-time (e.g., max_position_size > 0, allowed_markets non-empty)
   - Show concrete examples in placeholders (e.g., "e.g., BTC-USDC, ETH-USDC" for allowed_markets)
   - Display field type and constraints clearly (e.g., "uint256 (USDC wei)" or "0-100 (percent)")
   - Include "Copy Example Configuration" button to pre-fill with realistic sample data

3. **Policy Hash Computation**
   - Once user fills and submits the form, compute policy_hash = keccak256(abi.encode(...))
   - Use ethers.js: ethers.AbiCoder.defaultAbiCoder().encode([type1, type2, ...], [value1, value2, ...])
   - Policy hash must be deterministic: same inputs always produce same hash
   - Display policy_hash in a read-only, copyable field
   - Show policy_nonce (default: 0) and policy_expiry (default: current_timestamp + 10 minutes)
   - Include a "Test Hash Determinism" button that re-computes and confirms hash stability

4. **Enforcement Stack Visualization**
   - Display three-tier enforcement architecture:
     - **Tier 1 - PolicyEngine (Control Plane)**: List specific validation checks (e.g., "Position size < max_position_size_usdc", "Market in allowlist")
     - **Tier 2 - PolicyValidator (On-Chain)**: Proof verification, replay prevention via nonces/expiry, oracle-gated checks
     - **Tier 3 - Crossmint Agent Wallet**: Smart contract guardrails, spend limits, recipient whitelisting
   - Use color-coded badges for each tier
   - Explain in plain language what each tier does and why it matters
   - Include a collapsible "Technical Details" section for each tier

5. **API Integration (Backend)**
   - POST /api/agents/create: Save agent config to agents table, derive policy_hash, return agent_id and tx_hash
   - POST /api/agents/validate: Validate policy_hash against stored config (idempotent)
   - GET /api/agents/:agent_id: Retrieve full agent status, balance, recent actions
   - POST /api/agents/:agent_id/policy: Update policy, store old and new hashes for audit trail
   - All responses include signed policy_proof once control plane signs it (async)
   - Include proper error handling with descriptive messages

6. **Claude Tool Schema Display**
   - Show the Claude tool schema (JSON) for the selected agent type
   - Display in a syntax-highlighted code block (read-only)
   - Include a sidebar explaining:
     - What each tool does
     - Tool input schema (parameters, types, constraints)
     - Example usage
     - When this tool is called (ReAct loop condition)
   - "Copy Tool Schema" button for developer reference

7. **Summary Screen Before Deployment**
   - Show all user inputs in a read-only summary
   - Display computed policy_hash with visual confirmation
   - Show estimated monthly costs (if applicable)
   - Display all three enforcement tiers and what they validate
   - Prominent "Deploy Agent" button
   - "Edit" button to go back and modify inputs

8. **Deployment & Feedback**
   - Show deployment progress:
     - Hashing inputs → policy_hash computed
     - Signing proof → proof generated
     - Submitting UserOperation → awaiting confirmation
     - Agent registered → ERC-8004 registry entry
   - Display final agent_id and tx_hash once deployed
   - Show "View on Explorer" link to on-chain transaction
   - Offer "Deploy Another Agent" or "View Agent Dashboard" options

Database Schema:
```sql
CREATE TABLE agents (
  agent_id BYTEA PRIMARY KEY,
  owner_address VARCHAR(42),
  agent_type VARCHAR(20),
  agent_name VARCHAR(64),
  agent_description VARCHAR(256),
  status VARCHAR(20),
  capital_allocation BIGINT,
  balance_usdc BIGINT,
  policy_hash BYTEA,
  tx_hash VARCHAR(66),
  created_at BIGINT,
  updated_at BIGINT,
  deployed_at BIGINT
);

CREATE TABLE agent_policies (
  policy_id SERIAL PRIMARY KEY,
  agent_id BYTEA REFERENCES agents(agent_id),
  policy_parameters JSONB,
  policy_hash BYTEA,
  nonce INT DEFAULT 0,
  expiry BIGINT,
  is_active BOOLEAN DEFAULT TRUE,
  created_at BIGINT
);

CREATE TABLE agent_actions (
  action_id BYTEA PRIMARY KEY,
  agent_id BYTEA REFERENCES agents(agent_id),
  action_type VARCHAR(20),
  timestamp BIGINT,
  amount_usdc BIGINT,
  counterparty VARCHAR(42),
  tx_hash VARCHAR(66),
  policy_proof_hash BYTEA,
  status VARCHAR(20),
  created_at BIGINT
);

CREATE TABLE agent_memory (
  memory_id SERIAL PRIMARY KEY,
  agent_id BYTEA REFERENCES agents(agent_id),
  observation_type VARCHAR(50),
  embedding VECTOR(1536),
  content TEXT,
  created_at BIGINT,
  INDEX ON (agent_id, observation_type)
);
```
