=========================================================================
INTEGRATION PRINCIPLES (these define "production" here)
=========================================================================
1. ACTOR = SESSION. The current member is whoever is authenticated. The platform never sends
   an actor field on session-authenticated calls (core strips it anyway). There is NO member
   switcher of any kind — demoing two approvers means two signed-in sessions.
2. ENFORCEMENT LIVES IN CORE ONLY. Do not port, keep, or re-implement authorizeApproval or the
   checks client-side. Delete any mock gate. UI affordances (hiding approve controls from a
   viewer, disabling above-limit approvals) are derived from the CURRENT MEMBER'S envelope
   fetched from core — conveniences only; the server's response is always the final word. A
   user who bypasses the UI still hits core's 403.
3. REJECTIONS ARE CORE'S. Every 403 { error, reason, detail } renders as an inline #D20344
   notice. See the reason-mapping section below.
4. STATE COMES FROM CORE. Proposal status (including "awaiting_second_approval"), the approvals
   array, actor verification — all read from core responses/webhooks; the platform holds no
   parallel state machine. On network failure, show last known state + retry; never
   optimistically mark approved before core confirms.

- Webhooks/subscriptions: member.changed → refresh members views;
  proposal.awaiting_second_approval → update the queue row live.
- Every member mutation confirms in plain language BEFORE the call, then reflects core's
  response — including surfacing last_admin_protected as a disabled control with caption "At
  least one admin is required", and proactively disabling that control when the fetched member
  list shows only one active admin.

REJECTION-REASON MAPPING (all seven; plain-language, from core's exact reason string)
=========================================================================
Map each reason to one sentence. IMPORTANT: self_approval_blocked now has TWO distinct cases,
disambiguated by detail.payee_unresolved — render them differently:
- actor_unresolved       → "We couldn't confirm who you are for this approval. Sign in again."
- actor_inactive         → "Your account is inactive. An admin needs to reactivate you."
- domain_not_authorized  → "You're not authorized to approve this type of payment."
- actor_limit_exceeded   → "This amount is above your approval limit ({limit}). It needs a
                            higher-authority approver." (pull limit from detail if present)
- second_approval_required → "You've already approved this — a second, different approver is
                            required."
- self_approval_blocked WITH detail.payee_unresolved === true
                         → "We couldn't verify the recipient, so this needs manual review
                            before approval." (NOT phrased as self-approval — the block is a
                            precaution, the user may not be the payee at all)
- self_approval_blocked WITHOUT payee_unresolved
                         → "You can't approve a payment to yourself. A different approver is
                            required."
- last_admin_protected   → "At least one admin is required — you can't remove the last one."
- unknown/unlisted reason → generic "Not permitted." + raw reason shown in Technical Detail.
Never a silent no-op, never a dead tap.

SURFACES (same design as specced; now fed by core)
=========================================================================
1) MEMBERS PAGE — Settings → Team. Kicker "Your Team", Title "Who can approve, and up to how
   much.", Text "Authority is enforced by Brain core on every surface — this page manages it."
   Rows: displayName, role chip, envelope in mono ("AP + payroll · up to $10,000"), active
   state, chevron. Read-only for non-admins (derived from session member's role).
2) MEMBER DETAIL popup: role, envelope (domains chips; perItemLimit inline-editable mono →
   PATCH on confirm), identityLinks, "Recent approvals" resolving into audit records via the
   existing open helpers. LOCKED rows rendered from policy, not hardcoded: "Members can never
   approve their own payments" and the tenant second-approval threshold ("Second approval above
   $X (tenant policy)") — read the threshold from core's policy surface; do not hardcode.
3) SECOND-APPROVAL FLOW: when an approve response returns status "awaiting_second_approval",
   the queue row enters the held state ("Awaiting second approval · 1 of 2 ·
   {approvals[0].email} approved") — NOT executing, NOT removed. Same member re-approving
   surfaces core's second_approval_required rejection. A distinct qualified member's session
   completes it; the lifecycle then shows TWO approval steps, each with actorRole and authority
   line populated from core data. Viewers see no approve controls and core rejects them
   regardless.
4) AUDIT INTEGRATION: lifecycle approval steps stamp actorRole + authority suffix from core's
   response; actorVerification renders muted in Technical Detail ("verified via session").
   member_change events from core appear in the Audit Log; linked members resolve via
   openMemberDetail — same resolver pattern as rules/vendors (canonical source is the
   API-backed members cache, resolve-or-warn, plain-text fallback for a deactivated/unknown
   member: "(member inactive)" — deactivated members must still resolve for historical
   records, per deactivate-not-delete).
